Motivo por el que hacer un blog

En la mayoría de los libros, posts y pequeños proyectos que he ido leyendo por internet sugerían la idea de hacer un blog. Estos coincidían en que escribir un post te obliga a sentarte y concentrarte en una idea que tienes que explicar lo suficientemente clara y concisa para que el resto la entienda. Hace que tus pensamientos se aclaren y tengas una comprensión mayor sobre ese conocimiento. Otro de los beneficios de hacerlo es que dejas por escrito un conocimiento que has adquirido para así, en un futuro recuperarlo si fuera el caso. Así, evitas volver a visitar los mismos foros, páginas web y libros que ya leíste en su momento puesto que has dejado escrita tu comprensión sobre algo en particular y los pasos a seguir para acometer su objetivo.

Por estos motivos, dejo aquí los pasos desde cero que he seguido para subir este blog a internet.

Conocimientos básicos antes de empezar

Para hacer la web me han sido necesarios una serie de conocimientos básicos sobre los programas que he utilizado para crearla. Dejo un listado de algunos de los libros imprescindibles:

  • R for Data Science, primer libro que hay que leer para aprender a utilizar R y RStudio. Esta orientado al Data Science pero los primeros pasos son muy útiles para aprender el funcionamiento de R y la instalación de paquetes.
  • blogdown: Creating Websites with R Markdown, todo lo que se necesita aprender a la hora de utilizar RStudio y el paquete blogdown está aquí. Sin embargo, los pasos a seguir que indican para crear un blog no me han sido de utilidad.
  • Github, es necesario tener una cuenta en Github que albergará el repositorio de la web. En Github se pueden encontrar muchos recursos para aprender Git y al crear la cuenta se hace una pequeña introducción al funcionamiento, como crear repositorios y como hacer los cambios.
  • Conocimientos muy básicos sobre la terminal (en Windows -> Símbolo del Sistema)
  • Tener instalado Git para usar los comandos en Git Bash o en el terminal

Primer paso: Github

  1. El primer paso es crear un repositorio en Github. Para ello, entramos en Github y creamos una cuenta. Una vez que la tenemos nos aparecerá la opción de Start a project y redirigirá a los primeros pasos para crear el repositorio. Aquí rellenamos los cuadros de diálogo con el nombre de la carpeta y una pequeña descripción de su contenido. Es importante marcar la casilla de README.
  1. Una vez en la página principal del nuevo repositorio, vamos a la pestaña verde Clone or download y copiamos en el portapapeles la url del repositorio.

Segundo paso: Terminal

  1. Abrimos el terminal de Windows (Símbolo del Sistema) y nos deplazamos con el comando cd a la carpeta del ordenador donde queremos que esten los archivos físicos de la web.

  2. Importante: Con Git instalado en el ordenador y el terminal ejecutandose en la carpeta elegida de nuestro ordenador escribimos el siguiente comando:

> git clone [url de nuestro repositorio]

Con este comando se clona el repositorio que hemos creado en Github a nuestra carpeta y nos permitirá tenerlo actualizado a medida que vayamos progresando con la página web.

{{% alert info %}} NOTA: Al elegir la carpeta donde quieres tener el repositorio de Github hay que tener en cuenta que cuando se ejecuta el comando git clone se descarga una carpeta (el repositorio), esto significa que si tus archivos los quieres tener en la ruta (ejemplo) C:\Users\Alberto\Desktop\mydeeplearning, git clone lo tienes que ejecutar en la ruta C:\Users\Alberto\Desktop {{% /alert %}}

Tercer paso: RStudio

  1. Abrimos RStudio e instalamos blogdown o en su defecto devtools si no lo tienes instalado ya. Con el siguiente comando se instalarán los paquetes que falten de devtools o devtools al completo. Después de esto, instalamos Hugo.
# Instalación devtools
if (!requireNamespace("devtools")) install.packages("devtools")

# Instalación Hugo
blogdown::install_hugo()

  1. Creamos un nuevo proyecto. En la barra de herramientas pinchamos en File - New Project... - Existing Repository y una vez aquí buscamos la ruta de la carpeta donde tenemos nuestro repositorio.

  2. Una vez dentro de nuestro proyecto, vamos a la ventana de la interfaz de RStudio donde se encuentra Files y abrimos el archivo .gitignore que se encontrará en la carpeta principal del repositorio. Este archivo tendrá por defecto una serie de parámetros a los cuales nosotros le añadiremos Thumbs.db y public/. Las líneas de código del archivo deberán quedar así:

.Rproj.user
.Rhistory
.RData
.Ruserdata
blogdown
Thumbs.db
public/

{{% alert info %}} NOTA: antes de hacerlo es importante saber varias cosas:

  • Hay que elegir el estilo de la página. Este paso es fundamental puesto que una vez que elijas el tema estético de la web, este no podrá ser cambiado. Por tanto, hay que tomarse unos minutos en buscar y elegir el adecuado a tus gustos en la página Hugo themes. El tema condicionará el funcionamiento de la web, el config.toml y las características que se le quieran añadir. Estos parámetros no suelen ser equivalentes entre temas por lo que normalmente no será posible el cambio de un tema a otro sin perder las modificaciones que se le hayan hecho. A la hora de elegir el tema también hay que tener en cuenta las actualizaciones que se le realizan. Esto significa que si el repositorio del tema no ha sido actualizado en mucho tiempo, es que no tiene un desarollo continuo y lo más probable es que tenga fallos de funcionamiento que no han sido reparados. Para instalar el tema elegido, al comando de creación de la página habrá que añadirle la ruta del repositorio de Github donde se encuentra el tema. (Pongo como ejemplo el tema que he elegido yo).

  • Es posible ejecutar el comando a secas, blogdown::new_site(), siendo entonces el tema por defecto el que se instalará, lithium. {{% /alert %}}

blogdown::new_site(theme = "kakawait/hugo-tranquilpeak-theme")
  1. Editar las opciones del proyecto. El siguiente paso es editar las opciones del proyecto para especificar que en lo que se está trabajando es una web. Para ello vamos a la barra de herramientas Tools - Project Options... y en la pestaña de Build Tools - Project build tools seleccionamos Website. Es importante también quitar el check de la casilla Preview site after building.
  1. Editar la configuración de la web. En este paso, entramos al config.toml donde están las características básicas de nuesto sitio web. El archivo config.toml es el primer documento que buscará Hugo para general la web. Es un documento importante que requiere tiempo por eso le voy a dedicar un post a parte.

Quinto paso: Los posts

{{% alert danger %}} NOTA: para este paso hay que tener un poco claro el flujo de trabajo y los Addins {{% /alert %}}

Simplemente hay que hacer click en el Addin New Post y rellenar los cuadros de diálogo con las características del nuevo post.

Por lo que respecta al último elemento del cuadro de diálogo, a la hora de elegir el formato de archivo del post es importante leer la información que se explica sobre ello en blogdown book chapter on output formats. Básicamente se explican las diferencias entre Markdown (.md) y Rmarkdown (.Rmd). Personalmente utilizo Rmarkdown para redactar los posts puesto que hace más fácil la tarea de incluir pedazos de código. Además, si el post que se está redactando tiene pedazos de código funcionales, es decir, que ejecutan una orden en R correctamente, se pueden introducir los resultados o gráficas fácilmente.

Sexto paso: Netlify

Finalmente lo último que queda por hacer es subir la web a Netlify. Para ello, entramos en Netlify y nos registramos a través de nuestra cuenta de Github. Después presionamos sobre la pestaña New site from Git y seguimos los pasos que se nos muestran para crear la web. Estos son los siguientes:

  1. En el panel de Continuous Deployment elegimos la opción de Github
  2. Tras conectar con nuestra cuenta de Github (paso que hará automáticamente o pedirá conectarse) seleccionamos el repositorio donde tenemos los archivos de la web
  3. Por último en las Building Options, en la opción Branch to deploy dejamos master por defecto. En cuanto a las opciones de construcción:
  • Build command: será el comando que utilice Netlify para generar la web. Por defecto podríamos poner hugo, sin embargo, esto generaría la página con la versión de hugo 0.17. Se puede especificar la versión de Hugo con la que se quiere que se genere la web. En mi caso, he generado con la versión 0.19, introduciendo en este apartado hugo_0.19.
  • Publish directory: este es el directorio sobre el que se genera la web y se deberá introducir ‘public’.

Finalmente ya solo queda ir a la página de configuración una vez generada la página y cambiar la url que se crea por defecto por el nombre que se desee, copiar dicha dirección y sustituirla en la línea de código del config.toml.

baseURL = "https://mydeeplearning.netlify.com/"

El flujo de trabajo y los Addins

El flujo de trabajo son los pasos que se suelen dar a la hora de mantener la web, continuar el desarrollo de un post en el que se está trabajando, asegurarte de que los cambios se producen correctamente y finalmente actualizar los cambios subiéndolos al reposiorio para dejar la página operativa. Los Addins son herramientas que ofrece RStudio para facilitarte este proceso. La interfaz del IDE de RStudio por lo general es muy intuitiva. Los Addins los encontraremos bajo la barra de herramientas, siendo las tres opciones que nos ofrece y que nos interesan para el blog las siguientes:

  • New Post: esta pestaña abrirá un panel en el que podremos rellenar el metadata del post, nos dará a elegir el formato del archivo del post(.md, .Rmd, .Rmarkdown) y creará el documento para comenzar a editarlo y empezar el post.
  • Serve Site: esta pestaña activará LiveReload, muy importante para el flujo de trabajo a la hora de escribir los posts. En esencia, lo que hace es generar la página en un servidor local de nuestro ordenador para la previsualización cada vez que guardamos un cambio en uno de los archivos del proyecto.
  • Update Metadata: esta pestaña abre una panel que nos permite cambiar el metadata de post sobre el que se está trabajando. Personalmente este Addin no lo utilizo porque cambio el metadata directamente en el script de los posts.

El flujo de trabajo con el que se suele funcionar es el siguiente:

  1. Abres el proyecto de la web en RStudio
  2. Activas el Addin Serve Site
  3. Abres el post sobre el que estás trabajando o creas uno nuevo
  4. Guardas los cambios las veces necesarias para visualizar el trabajo que has realizado (nunca se utiliza la opción knit). Cada vez que guardas los cambios el LiveReload mostrará el estado actual de la web en el paner Viewer
  5. Cuando estás satisfecho con el trabajo realizado, vas panel de la interfaz Environment y en la pestaña Git presionamos en Commit. Se seleccionan todos los cambios que se han realizado y los marcamos como Staged, escribimos el comentario de los cambios que hemos realizado y presionamos en Commit. Aparecerá un cuadro de dialogo que muestra el transcurso de las actividades y finalmente el resumen. Por último, presionamos en Push para actualizar los cambios en el repositorio. Esto equivale a los comandos git add/commit/push que habría que haber introducido en el terminal del sistema dentro de la carpeta de la web en nuestro ordenador
